# 等待时间计算模式配置 - 用户指南

## 概述

现在用户可以**自定义等待时间的计算和显示方式**，根据不同的分析需求选择最合适的模式。

## 📍 配置位置

在左侧边栏的 **"Wait Time Calculation"** 部分，位于 **"Simulation Mode"** 之后。

```
左侧边栏
├─ Controls (控制)
├─ Business Time (工作时间)
├─ Auto Pause (自动暂停)
├─ Demand Peaks (需求峰值)
├─ Simulation Mode (模拟模式)
├─ Wait Time Calculation ← 新增
└─ AI Scenario (AI 场景)
```

## 🎯 三种模式

### 模式 1: 📊 Both (推荐)
**显示两种等待时间，提供完整分析**

```
选择此模式时显示：
┌────────────────────────────────┐
│ Queue Wait (Calendar)          │
│ 18.5 hours                     │
│                                │
│ Queue Wait (Working)           │
│ 2.3 hours                      │
│                                │
│ Diagnostic Working Wait        │
│ 3.1 hours                      │
└────────────────────────────────┘
```

**适用场景**：
- ✅ 需要全面分析
- ✅ 同时关注客户体验和内部效率
- ✅ 做决策时需要对比两个视角
- ✅ 首次使用系统（推荐默认）

**优点**：
- 信息最完整
- 可以看出非工作时间的影响
- 便于识别是资源问题还是工作时间问题

**缺点**：
- UI 卡片较多
- 如果只关注一个视角可能显得冗余

---

### 模式 2: 📅 Calendar Time
**仅显示日历等待时间（包含非工作时间）**

```
选择此模式时显示：
┌────────────────────────────────┐
│ Queue Wait                     │
│ 18.5 hours                     │
└────────────────────────────────┘
```

**定义**：从入队到处理的完整日历时间

**适用场景**：
- ✅ 客户 SLA 报告
- ✅ 对外承诺跟踪
- ✅ 响应时间监控
- ✅ 客户体验分析

**使用案例**：
```
场景：承诺客户 "24 小时内响应"

✓ 使用 Calendar Time
  - 客户不关心你的工作时间
  - 他们只知道从提交到回复等了多久
  
示例：
  周五 17:00 提交 → 周一 9:00 回复
  Calendar Wait = 64 小时 ❌ 违反 SLA
```

**何时选择**：
- 向客户报告时
- 评估客户满意度时
- 设定或检查 SLA 时
- 做市场对外宣传时

---

### 模式 3: ⏱️ Working Time
**仅显示工作等待时间（排除非工作时间）**

```
选择此模式时显示：
┌────────────────────────────────┐
│ Queue Wait                     │
│ 2.3 hours                      │
└────────────────────────────────┘
```

**定义**：仅计算工作时间内的队列等待

**统计口径**：`Queue Wait (Working)` 按完成 item 的真实经历聚合（item-weighted）。如果一个步骤很少被经过，它不会和高流量步骤拥有同等权重。

**诊断补充**：面板中还会保留 `Diagnostic Working Wait`，这是步骤级平均值，用于找瓶颈；它不代表平均每个 item 的真实等待体验。

**适用场景**：
- ✅ 队列效率分析
- ✅ 容量规划
- ✅ 资源优化决策
- ✅ 内部运营改进

**使用案例**：
```
场景：决定是否增加客服人员

✓ 使用 Working Time
  - 非工作时间的等待增加人员也解决不了
  - 只有工作时间的队列积压才需要更多资源
  
示例：
  周五 17:00 入队 → 周一 9:00 处理
  Working Wait = 0 小时 ✅ 队列效率很好
  
决策：不需要增加人员，考虑延长工作时间
```

**何时选择**：
- 评估是否需要增加资源时
- 优化队列处理效率时
- 对比不同工作时间配置时
- 做内部绩效评估时

---

## 🔄 如何切换模式

### 步骤 1：打开左侧边栏
点击左上角的菜单图标或按快捷键。

### 步骤 2：找到 "Wait Time Calculation"
向下滚动，在 "Simulation Mode" 下方。

### 步骤 3：点击选择模式
三个按钮：
- **📊 Both (Recommended)** - 绿色背景表示当前选中
- **📅 Calendar Time** - 琥珀色
- **⏱️ Working Time** - 黄色

### 步骤 4：查看变化
统计面板会立即更新，显示相应的等待时间指标。

## 💡 实际使用建议

### 场景 1：首次使用 / 学习阶段
**推荐模式**：📊 **Both**

**理由**：
- 看到两个指标的差异，理解概念
- 发现哪个指标对你的业务更重要
- 之后可以切换到单一模式

---

### 场景 2：准备客户报告
**推荐模式**：📅 **Calendar Time**

**工作流程**：
1. 切换到 Calendar Time 模式
2. 截图或导出统计数据
3. 用于客户报告、SLA 检查
4. 完成后切回 Both 模式

**示例报告**：
```
客户服务 SLA 报告
────────────────────
平均响应时间：18.5 小时
SLA 目标：24 小时
状态：✅ 达标
```

---

### 场景 3：内部运营会议
**推荐模式**：⏱️ **Working Time**

**工作流程**：
1. 切换到 Working Time 模式
2. 分析队列效率
3. 决定资源配置
4. 讨论完成后切回 Both

**示例讨论**：
```
运营效率分析
────────────────────
队列等待（工作时间）：2.3 小时
目标：< 2 小时
状态：⚠️ 略高于目标

建议：增加 1 名处理人员
```

---

### 场景 4：容量规划
**推荐模式**：📊 **Both** → ⏱️ **Working Time**

**决策流程**：
```
1. 先看 Both 模式
   Calendar Wait = 20h
   Working Wait = 2h
   
2. 分析差异
   差值 = 18h = 非工作时间
   
3. 切换到 Working Time 聚焦分析
   Working Wait = 2h < 目标 3h
   
4. 决策
   ✅ 队列效率良好，无需增加资源
   ⚠️ 如需改善客户体验，考虑延长工作时间
```

---

### 场景 5：对比不同配置
**推荐模式**：⏱️ **Working Time**

**实验流程**：
```
配置 A：周一至周五 9:00-17:00
切换到 Working Time 模式
结果：Working Wait = 2h

配置 B：周一至周五 9:00-21:00
Working Time 模式不变
结果：Working Wait = 1.5h

对比：延长工作时间减少了 25% 的队列等待
```

**为什么用 Working Time**：
- Calendar Time 会因工作时间长度不同而不公平
- Working Time 只看队列效率，公平对比

## 🎓 常见问题

### Q1: 默认是哪个模式？
**A**: 📊 **Both** (推荐模式)

系统默认显示两个指标，让用户了解完整情况。

---

### Q2: 切换模式会影响数据吗？
**A**: ❌ **不会**

切换模式**只影响显示**，不影响计算：
- 后台始终计算两种等待时间
- 切换只是改变 UI 显示哪些卡片
- 数据本身不会改变

---

### Q3: 模式选择会保存吗？
**A**: ⚠️ **当前不会**（可能后续更新）

目前模式选择在以下情况会重置：
- 刷新页面
- 重新打开应用

**临时解决方案**：
每次打开应用后，快速切换到你喜欢的模式（只需 1 秒）。

---

### Q4: 哪个模式最准确？
**A**: ✅ **都准确，只是视角不同**

- Calendar Time = 客户视角（完整等待）
- Working Time = 运营视角（队列效率）
- 没有谁更准确，只有哪个更适合你的场景

类比：
```
看一个人：
- 身高 = Calendar Time（完整高度）
- 净高 = Working Time（去掉鞋跟）
都是真实的，只是测量方式不同
```

---

### Q5: 两个指标差异很大，正常吗？
**A**: ✅ **完全正常**

**示例 1：周末效应大**
```
周五 17:00 入队 → 周一 9:00 处理
Calendar Wait = 64h
Working Wait = 0h
差异 = 64h（整个周末）
```

**示例 2：工作日内等待**
```
周一 10:00 入队 → 周一 12:00 处理
Calendar Wait = 2h
Working Wait = 2h
差异 = 0h（全在工作时间内）
```

**规律**：
- 差异大 = 非工作时间影响大
- 差异小 = 主要是队列积压

---

### Q6: 如何知道该用哪个模式？
**A**: 看你在回答什么问题

| 问题 | 用哪个模式 |
|------|-----------|
| 客户等了多久？ | 📅 Calendar |
| 队列效率如何？ | ⏱️ Working |
| 是否需要加人？ | ⏱️ Working |
| SLA 是否违反？ | 📅 Calendar |
| 哪个配置更高效？ | ⏱️ Working |
| 不确定 / 学习中 | 📊 Both |

## 🚀 高级用法

### 技巧 1：快速切换对比
在分析时，可以快速切换模式对比：
1. Both 模式看全局
2. Calendar 模式专注客户视角
3. Working 模式专注运营视角

**键盘快捷键**（如果实现）：
- `Ctrl+1` = Both
- `Ctrl+2` = Calendar
- `Ctrl+3` = Working

---

### 技巧 2：分时段分析
不同时间段用不同模式：

**早上**：Both 模式，查看过夜情况
**工作中**：Working 模式，监控队列效率
**准备报告**：Calendar 模式，客户视角

---

### 技巧 3：团队协作
不同角色用不同模式：

| 角色 | 推荐模式 | 原因 |
|------|----------|------|
| 客服主管 | 📅 Calendar | 关注客户体验 |
| 运营经理 | ⏱️ Working | 优化内部效率 |
| 数据分析师 | 📊 Both | 全面分析 |
| CEO | 📅 Calendar | 对外承诺 |

## 📝 总结

### 核心原则
> **右键选择工具，选择适合你当前任务的模式**

### 快速决策树
```
你在做什么？
├─ 给客户看 → 📅 Calendar Time
├─ 内部优化 → ⏱️ Working Time
└─ 不确定 → 📊 Both (Recommended)
```

### 记住这点
**没有"最好"的模式，只有"最适合"的模式**

根据你的具体任务选择，必要时随时切换！
