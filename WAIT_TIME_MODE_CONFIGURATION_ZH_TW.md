# 等待時間計算模式配置 - 使用者指南

## 概述

現在使用者可以**自訂等待時間的計算和顯示方式**，根據不同的分析需求選擇最合適的模式。

## 📍 設定位置

在左側邊欄的 **"Wait Time Calculation"** 部分，位於 **"Simulation Mode"** 之後。

```
左側邊欄
├─ Controls (控制)
├─ Business Time (工作時間)
├─ Auto Pause (自動暫停)
├─ Demand Peaks (需求峰值)
├─ Simulation Mode (模擬模式)
├─ Wait Time Calculation ← 新增
└─ AI Scenario (AI 場景)
```

## 🎯 三種模式

### 模式 1: 📊 Both (推薦)
**顯示兩種等待時間，提供完整分析**

```
選擇此模式時顯示：
┌────────────────────────────────┐
│ Queue Wait (Calendar)          │
│ 18.5 hours                     │
│                                │
│ Queue Wait (Working)           │
│ 2.3 hours                      │
│                                │
│ Diagnostic Working Wait        │
│ 3.1 hours                      │
└────────────────────────────────┘
```

**適用場景**：
- ✅ 需要全面分析
- ✅ 同時關注客戶體驗和內部效率
- ✅ 做決策時需要對比兩個視角
- ✅ 首次使用系統（推薦預設）

**優點**：
- 資訊最完整
- 可以看出非工作時間的影響
- 便於識別是資源問題還是工作時間問題

**缺點**：
- UI 卡片較多
- 如果只關注一個視角可能顯得冗餘

---

### 模式 2: 📅 Calendar Time
**僅顯示日曆等待時間（包含非工作時間）**

```
選擇此模式時顯示：
┌────────────────────────────────┐
│ Queue Wait                     │
│ 18.5 hours                     │
└────────────────────────────────┘
```

**定義**：從入隊到處理的完整日曆時間

**適用場景**：
- ✅ 客戶 SLA 報告
- ✅ 對外承諾追蹤
- ✅ 回應時間監控
- ✅ 客戶體驗分析

**使用案例**：
```
場景：承諾客戶「24 小時內回應」

✓ 使用 Calendar Time
  - 客戶不關心你的工作時間
  - 他們只知道從提交到回覆等了多久
  
範例：
  週五 17:00 提交 → 週一 9:00 回覆
  Calendar Wait = 64 小時 ❌ 違反 SLA
```

**何時選擇**：
- 向客戶報告時
- 評估客戶滿意度時
- 設定或檢查 SLA 時
- 做市場對外宣傳時

---

### 模式 3: ⏱️ Working Time
**僅顯示工作等待時間（排除非工作時間）**

```
選擇此模式時顯示：
┌────────────────────────────────┐
│ Queue Wait                     │
│ 2.3 hours                      │
└────────────────────────────────┘
```

**定義**：僅計算工作時間內的佇列等待

**統計口徑**：`Queue Wait (Working)` 按完成 item 的真實經歷聚合（item-weighted）。如果某個步驟很少被經過，它不會和高流量步驟擁有同等權重。

**診斷補充**：面板中也會保留 `Diagnostic Working Wait`，這是步驟級平均值，用於找瓶頸；它不代表平均每個 item 的真實等待體驗。

**適用場景**：
- ✅ 佇列效率分析
- ✅ 容量規劃
- ✅ 資源優化決策
- ✅ 內部營運改進

**使用案例**：
```
場景：決定是否增加客服人員

✓ 使用 Working Time
  - 非工作時間的等待增加人員也解決不了
  - 只有工作時間的佇列積壓才需要更多資源
  
範例：
  週五 17:00 入隊 → 週一 9:00 處理
  Working Wait = 0 小時 ✅ 佇列效率很好
  
決策：不需要增加人員，考慮延長工作時間
```

**何時選擇**：
- 評估是否需要增加資源時
- 優化佇列處理效率時
- 對比不同工作時間配置時
- 做內部績效評估時

---

## 🔄 如何切換模式

### 步驟 1：開啟左側邊欄
點擊左上角的選單圖示或按快捷鍵。

### 步驟 2：找到「Wait Time Calculation」
向下捲動，在「Simulation Mode」下方。

### 步驟 3：點擊選擇模式
三個按鈕：
- **📊 Both (Recommended)** - 綠色背景表示目前選中
- **📅 Calendar Time** - 琥珀色
- **⏱️ Working Time** - 黃色

### 步驟 4：查看變化
統計面板會立即更新，顯示相應的等待時間指標。

## 💡 實際使用建議

### 場景 1：首次使用 / 學習階段
**推薦模式**：📊 **Both**

**理由**：
- 看到兩個指標的差異，理解概念
- 發現哪個指標對你的業務更重要
- 之後可以切換到單一模式

---

### 場景 2：準備客戶報告
**推薦模式**：📅 **Calendar Time**

**工作流程**：
1. 切換到 Calendar Time 模式
2. 截圖或匯出統計資料
3. 用於客戶報告、SLA 檢查
4. 完成後切回 Both 模式

**範例報告**：
```
客戶服務 SLA 報告
────────────────────
平均回應時間：18.5 小時
SLA 目標：24 小時
狀態：✅ 達標
```

---

### 場景 3：內部營運會議
**推薦模式**：⏱️ **Working Time**

**工作流程**：
1. 切換到 Working Time 模式
2. 分析佇列效率
3. 決定資源配置
4. 討論完成後切回 Both

**範例討論**：
```
營運效率分析
────────────────────
佇列等待（工作時間）：2.3 小時
目標：< 2 小時
狀態：⚠️ 略高於目標

建議：增加 1 名處理人員
```

---

### 場景 4：容量規劃
**推薦模式**：📊 **Both** → ⏱️ **Working Time**

**決策流程**：
```
1. 先看 Both 模式
   Calendar Wait = 20h
   Working Wait = 2h
   
2. 分析差異
   差值 = 18h = 非工作時間
   
3. 切換到 Working Time 聚焦分析
   Working Wait = 2h < 目標 3h
   
4. 決策
   ✅ 佇列效率良好，無需增加資源
   ⚠️ 如需改善客戶體驗，考慮延長工作時間
```

---

### 場景 5：對比不同配置
**推薦模式**：⏱️ **Working Time**

**實驗流程**：
```
配置 A：週一至週五 9:00-17:00
切換到 Working Time 模式
結果：Working Wait = 2h

配置 B：週一至週五 9:00-21:00
Working Time 模式不變
結果：Working Wait = 1.5h

對比：延長工作時間減少了 25% 的佇列等待
```

**為什麼用 Working Time**：
- Calendar Time 會因工作時間長度不同而不公平
- Working Time 只看佇列效率，公平對比

## 🎓 常見問題

### Q1: 預設是哪個模式？
**A**: 📊 **Both** (推薦模式)

系統預設顯示兩個指標，讓使用者了解完整情況。

---

### Q2: 切換模式會影響資料嗎？
**A**: ❌ **不會**

切換模式**只影響顯示**，不影響計算：
- 後台始終計算兩種等待時間
- 切換只是改變 UI 顯示哪些卡片
- 資料本身不會改變

---

### Q3: 模式選擇會儲存嗎？
**A**: ⚠️ **目前不會**（可能後續更新）

目前模式選擇在以下情況會重設：
- 重新整理頁面
- 重新開啟應用程式

**臨時解決方案**：
每次開啟應用程式後，快速切換到你喜歡的模式（只需 1 秒）。

---

### Q4: 哪個模式最準確？
**A**: ✅ **都準確，只是視角不同**

- Calendar Time = 客戶視角（完整等待）
- Working Time = 營運視角（佇列效率）
- 沒有誰更準確，只有哪個更適合你的場景

類比：
```
看一個人：
- 身高 = Calendar Time（完整高度）
- 淨高 = Working Time（去掉鞋跟）
都是真實的，只是測量方式不同
```

---

### Q5: 兩個指標差異很大，正常嗎？
**A**: ✅ **完全正常**

**範例 1：週末效應大**
```
週五 17:00 入隊 → 週一 9:00 處理
Calendar Wait = 64h
Working Wait = 0h
差異 = 64h（整個週末）
```

**範例 2：工作日內等待**
```
週一 10:00 入隊 → 週一 12:00 處理
Calendar Wait = 2h
Working Wait = 2h
差異 = 0h（全在工作時間內）
```

**規律**：
- 差異大 = 非工作時間影響大
- 差異小 = 主要是佇列積壓

---

### Q6: 如何知道該用哪個模式？
**A**: 看你在回答什麼問題

| 問題 | 用哪個模式 |
|------|-----------|
| 客戶等了多久？ | 📅 Calendar |
| 佇列效率如何？ | ⏱️ Working |
| 是否需要加人？ | ⏱️ Working |
| SLA 是否違反？ | 📅 Calendar |
| 哪個配置更高效？ | ⏱️ Working |
| 不確定 / 學習中 | 📊 Both |

## 🚀 進階用法

### 技巧 1：快速切換對比
在分析時，可以快速切換模式對比：
1. Both 模式看全局
2. Calendar 模式專注客戶視角
3. Working 模式專注營運視角

**鍵盤快捷鍵**（如果實作）：
- `Ctrl+1` = Both
- `Ctrl+2` = Calendar
- `Ctrl+3` = Working

---

### 技巧 2：分時段分析
不同時間段用不同模式：

**早上**：Both 模式，查看過夜情況
**工作中**：Working 模式，監控佇列效率
**準備報告**：Calendar 模式，客戶視角

---

### 技巧 3：團隊協作
不同角色用不同模式：

| 角色 | 推薦模式 | 原因 |
|------|----------|------|
| 客服主管 | 📅 Calendar | 關注客戶體驗 |
| 營運經理 | ⏱️ Working | 優化內部效率 |
| 資料分析師 | 📊 Both | 全面分析 |
| CEO | 📅 Calendar | 對外承諾 |
